Unit Testing

Beim Unit Testing wollen wir nicht die einer Funktion bergebenen Argumente, sondern die Funktion selbst testen. Also im Zweifel ihren Rckgabewert.

Zwei Funktionen

  • Richtige Funktion 
my_sum <- function(x) {
    value <- 0
    for (x_i in x) {
        value <- value + x_i
    }
    return(value)
}
  • Falsche Funktion 
not_my_sum <- function(x) {
    value <- 0
    for (x_i in x) {
        value <- value - x_i
    }
    return(value)
}

Erwartungen

Die kleinste Testeinheit ist immer ein Vergleich.

In base R:

my_sum(c(2, 7)) == 9
## [1] TRUE
# better use identical(), see ?Comparison
identical(not_my_sum(c(2, 7)), 9)
## [1] FALSE

Code Coverage

Um zu sehen, ob Deine Unit Tests die Funktion gut testen, kannst Du ihre berdeckung berechnen. covr ist eines der Pakete, die Code Coverage implentieren und das einzige, das ich nutze. Ich wei es nicht, glaube aber, dass covr die branch coverage berechnet. Prinzipiell ist eine hohe berdeckung gut.

Wenn Du eine Datei mit einer Funktionsdefinition

src/my_sum_assertion.R
my_sum <- function(x) {
    assertthat::assert_that(is.numeric(x))
    value <- 0
    for (x_i in x) {
        value <- value + x_i
    }
    return(value)
}

und eine mit dem Test

src/test_my_sum_assertion.R
testthat::test_that("Simple test on summation.",
                    testthat::expect_identical(my_sum(c(2, 7)), 9)
                    )

hast, kannst Du die berdeckung berechnen.

cov <- covr::file_coverage(file.path("src", "my_sum_assertion.R"),
                           file.path("src", "test_my_sum_assertion.R"))
print(cov)
## Coverage: 100.00%

## src/my_sum_assertion.R: 100.00%

Wenn Du eine Funktion schreibst, in der eine Zeile eingebaut ist, die der Test nicht erreicht, weist Dich covr darauf hin:

src/my_sum_assertion.R
my_sum <- function(x) {
    assertthat::assert_that(is.numeric(x))
    if (! is.numeric(x)) message("You'll never get here!")
    value <- 0
    for (x_i in x) {
        value <- value + x_i
    }
    return(value)
}
cov <- covr::file_coverage(file.path("src", "my_sum_coverage.R"),
                           file.path("src", "test_my_sum_assertion.R"))
print(cov)
## Coverage: 83.33%

## src/my_sum_coverage.R: 83.33%

covr sagt Dir sogar, dass es die dritte Zeile ist:

covr::zero_coverage(cov)

## 2 src/my_sum_coverage.R    my_sum    3     0

Nebenwirkungen testen

Funktionen sind relativ einfach zu testen, wenn ihr Rckgabewert ihr Zweck ist. Wenn sie aber Nebenwirkungen bezwecken, finde ich Tests schwieriger zu schreiben.

Stell Dir vor, Du httest die fehlerhafte Lschfunktion aus Arbeiten mit Gltigkeitsbereichen in eine Datei geschrieben:

    objects <- ls(name = environment, all.names = TRUE)
    rm(list = objects)
    return(invisible(objects))

Nun schreibst Du eine Testdatei:

Was soll die ganze Testerei?

Stell Dir vor, Du erstelltest fter Abbildungen. Jetzt bist Du ein Mensch mit einem Gefhl fr Harmonien, daher willst Du gerne, wenn mglich, den Goldenen Schnitt bei der Wahl der Zuschnitte bercksichtigen.

Wenn Du Graphiken nach dem Goldenen Schnitt proportionieren willst, musst Du bei gegebener Lnge der kurzen Seite diese mit dem Goldenen Schnitt multipizieren um die Lnge der langen Seite zu erhalten.

Das ist eigentlich nicht allzu kompliziert, dennoch kommst Du manchmal durcheinander, also schreibst Du die folgende Funktion:

src/golden_rectangle.R
golden_rectangle  <- function(b) {
    a <- b * golden_ratio()
    return(c("a" = a, "b" = b))
}

Auch fr diese Funktion schreibst Du einen Test:

testthat::test_that("Test on golden rectangle", {

Jetzt kannst Du entweder eine neue Funktion schreiben oder die alte erweitern. Da die Funktionalitten zum selben Thema passen und golden_rectangle bislang recht simpel ist, schreibst Du Deine Funktion also um:

src/golden_rectangle.R
golden_rectangle  <- function(value, landscape = TRUE) {
    phi <- golden_ratio()
    if (isTRUE(landscape)) {
        a  <- value
        b <- a / phi
    } else {
        b <- value
        a <- b * phi
    }
    return(c("a" = a, "b" = b))
}

Herzlichen Glckwunsch, Du hast gerade die Schnittstelle Deiner Funktion zerschossen, indem Du ihr voreingestelltes Verhalten gendert hast. Glcklicherweise hast Du einen Test, der Dir jetzt um die Ohren fliegt:

covr::file_coverage(file.path("src", "golden_rectangle.R"),
                    file.path("src", "test_golden_rectangle.R"))
## Error: Test failed: 'Test on golden rectangle'
## * `result` not identical to `expectation`.
## 2/2 mismatches (average diff: 1)
## [1] 2.00 - 3.24 == -1.236
## [2] 1.24 - 2.00 == -0.764

Was bedeutet das? Du musst entweder

  • alle Aufrufe der Funktion, die Du vor der Vernderung ihrer Schnittstelle progammiert hast, anpassen (indem Du , landscape = FALSE einfgst) oder
  • die Funktion so ndern, dass der Test weiterhin funktioniert.

Du entscheidest Dich, die Funktion zu ndern (und erweiterst bei der Gelegenheit gleich den Rckgabewert):

src/golden_rectangle.R
golden_rectangle  <- function(value, landscape = FALSE) {
    phi <- golden_ratio()
    if (isTRUE(landscape)) {
        a  <- value
        b <- a / phi
    } else {
        b <- value
        a <- b * phi
    }
    return(c("a" = a, "b" = b))
}

Um Querformatige Proportionen zu erhalten musst Du nun aktiv Querformat einstellen: golden_rectangle(width, landscape = TRUE)

Der Test funktioniert wieder, da Du aber die Funktion gendert hast, berdeckt er sie nicht mehr vollstndig:
